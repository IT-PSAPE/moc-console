# MOC Broadcast: a public playlist player as a third app

MOC Console's "Broadcasts" section is a playlist *authoring* tool — staff assemble ordered cues (image/audio/video media) into playlists — but nothing can play those playlists anywhere. We are adding a third app, `apps/broadcast` (**MOC Broadcast**): a public, no-auth front-end where a viewer picks a workspace, browses its *published* playlists, and plays one on a continuous loop (foyer/auditorium display use case). "Broadcasting" is treated as a verb — making a playlist publicly playable — not a new domain entity; the thing played is still a `Playlist`.

## Considered options

- **Extend MOC Console with a public playback route.** Rejected: Console is wholly auth-gated (`RequireAuth`, workspace-scoped providers, `AppShell`). A public surface inside it would mean carving an exception through the auth shell and RLS model — more risk than a separate, intentionally-public app, which already has a proven template in MOC Request.
- **Introduce a `Broadcast` entity (playlist + share config + schedule window).** Rejected: nothing in the requirements needs an entity. It would add a table, mutations, and Console UI for a concept that is currently just "this playlist is public." Deferred until a real need (scheduling, per-broadcast analytics) appears.
- **Visibility via per-playlist share tokens** (mirroring the events `share/events/:token` pattern). Rejected for v1: the framing is "a public entry point for all the playlists" (a catalogue), not "send one link to one viewer." Tokens also need a table + generation + Console UI. `status = 'published'` already exists, is already settable in the editor, and matches the catalogue model. Tokens/unlisted can be layered on later without rework.
- **Baked `VITE_WORKSPACE_ID` per deploy** (the MOC Request convention). Rejected: the desired UX is a Netflix-account-style chooser — land, see the available workspaces, pick one. A baked workspace can't express that. The anon `list_signup_workspaces()` RPC (phase-22) already exposes id/name/slug without auth, so a runtime chooser needs no new backend.
- **Human-readable workspace slug in the URL** (`/sunday-service/...`). Rejected for v1: the user explicitly did not want the "slug situation." The workspace **id** is carried in the path instead — deep-linkable and refresh-safe without committing to slug-based routing.
- **Remove audio as a queueable cue type now** (audio only as background music). Deferred: it is a wider domain shift (type narrowing, Console editor hardening, a `queue` check constraint, data migration). The v1 player tolerates audio cues by rendering a minimal visual for the cue's duration.
- **Rename `videoSettings` → `playbackSettings`.** Deferred: it carries the `muted` flag the player reads, but renaming touches two apps for no functional gain in v1. Left as-is to limit blast radius.

## Consequences

- **`status = 'published'` is the public-visibility contract.** Phase-29 adds anon `SELECT` RLS on `playlists` (published only), `queue` (parent published), and `media` (referenced by a published playlist's queue or `music_id`). These are additive, anon-only policies that sit alongside the existing authenticated policies (RLS is OR-combined) and do not weaken them. Anyone can enumerate a workspace's *published* playlists — acceptable for a public display board; "unlisted" would need share tokens later.
- **New playback config is modelled now, implemented partially.** Phase-29 adds `thumbnail_url`, `playback_mode` (`loop|stop|sequence`) + `next_playlist_id`, `transition` (`cut|fade|crossfade`) + `transition_duration_ms`. The v1 player implements `loop` and `cut` only; `stop`/`sequence`/`fade`/`crossfade` are read but fall through (TODOs in `use-cue-player.ts` / `player.tsx`). Adding them later is a player change with zero schema churn.
- **Workspace identity is the id in the path.** Routes: `/` (chooser, anon RPC) → `/:workspaceId` (cinematic hero + rail) → `/:workspaceId/play/:playlistId` (player). No anon RLS was added to `public.workspaces`; the chooser's only workspace exposure is the existing SECURITY DEFINER RPC.
- **MOC Broadcast owns its own data layer**, consistent with ADR-0001's per-app `data/` precedent (`apps/broadcast/src/data/fetch-broadcast.ts`). It reuses `@moc/data` (Supabase client), `@moc/types` (Playlist/Cue/MediaItem + new `PlaybackMode`/`PlaylistTransition`), `@moc/ui`, `@moc/utils`.
- **The playback runtime is genuinely new code.** No media player existed; `apps/broadcast/src/features/player/` (cue iterator + renderers + background-music ducking + auto-hiding controls + Fullscreen API + autoplay-gesture fallback) has no analogue elsewhere in the repo to reuse.
- **Cover art is derived.** Playlists had no artwork; `thumbnail_url` is the new optional cover (pasted URL or upload via the existing `uploadMediaFile` pattern, reused as `uploadPlaylistThumbnail` into the same `media` bucket). When null, MOC Broadcast falls back to the first cue's media thumbnail, then a branded gradient.
- **Console editor gains playlist-level controls** (thumbnail, end behaviour, transition) in the existing detail-page accordion/strip — no new screens, reusing `@moc/ui` Input/Dropdown/Button primitives.
- **Phase files remain an intended-state snapshot, not a ledger.** Phase-29 is additive and idempotent; its header instructs verifying against the live Supabase before applying, since the live DB is authoritative.
- **Vercel needs a third project** with `rootDirectory` = `apps/broadcast` (its `vercel.json` already does SPA fallback). No `VITE_WORKSPACE_ID`; only `VITE_SUPABASE_URL` + publishable key.
